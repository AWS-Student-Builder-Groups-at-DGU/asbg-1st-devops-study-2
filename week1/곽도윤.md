# 1장: 쿠버네티스 입문 (Introduction to Kubernetes)

## 1. 컨테이너 오케스트레이션의 본질 (인프라 추상화)
* **기존 배포 방식의 한계**
  * VM 기반 배포는 오버헤드가 크고 리소스 낭비 심함.
  * 단일 Docker 배포는 멀티 노드 스케일링, 복구(Self-healing), 로드 밸런싱을 인프라 스크립트(Ansible, 쉘 스크립트 등)로 개발자가 직접 구현해야 해서 공수가 많이 듦.
* **쿠버네티스의 해결책**
  * 수많은 서버 인프라를 하나의 거대한 단일 컴퓨팅 리소스 풀로 추상화함.
  * 애플리케이션 배포, 네트워킹, 스케일링, 복구 작업을 엔진 레벨에서 자동화함.

## 2. 핵심 패러다임: 선언적 API & 조정 루프
* **선언적 API (Declarative API)**
  * 명령형(How) 배포가 아닌 최종 지향 상태(Desired State, What)를 YAML 파일로 선언하는 방식임.
  * 백엔드 코드로 치면 "컨테이너 3개를 항상 띄워라"라고 선언하면 끝임.
* **조정 루프 (Reconciliation Loop)**
  * `[현재 상태 관찰 -> 선언된 상태와 비교 -> 차이점 수정]` 과정을 무한 반복하는 이벤트 구동형 동기화 엔진임.
  * 특정 노드가 다운되어 컨테이너가 죽으면, 조정 루프가 이를 감지하고 다른 정상 노드에 자동으로 재배포(Self-healing)함.

## 3. 클러스터 아키텍처 (구조적 이해)
* 클러스터는 시스템 두뇌인 **제어 평면(Control Plane)**과 실제 워크로드가 도는 **워커 노드(Worker Node)**로 나뉨.



### A. 제어 평면 (Control Plane)
* **`kube-apiserver`**
  * 클러스터의 중심 관문이자 모든 내부 컴포넌트 및 외부 CLI(`kubectl`)의 유일한 REST API 엔드포인트임.
  * 무상태(Stateless)로 동작하여 수평 확장이 가능함.
* **`etcd`**
  * 클러스터의 모든 상태 데이터가 저장되는 고가용성 분산 Key-Value 저장소(Raft 합의 알고리즘 기반)임.
  * 백엔드 관점에서 **클러스터의 유일한 SOT(Source of Truth)**이므로, 데이터 정합성과 주기적인 백업이 매우 중요함.
* **`kube-scheduler`**
  * 컨테이너(Pod)의 리소스 요구량(CPU/Mem), 하드웨어 제약 조건 등을 고려하여 최적의 워커 노드를 필터링하고 할당하는 배치 엔진임.
* **`kube-controller-manager`**
  * 조정 루프를 실제로 실행하는 프로세스 모음(노드/레플리카/서비스 컨트롤러 등)임.

### B. 워커 노드 (Worker Node)
* **`kubelet`**
  * 각 노드에 상주하는 데몬 에이전트로, API 서버의 명령을 받아 컨테이너 런타임(CRI, 예: containerd)과 통신하며 컨테이너 라이프사이클을 실질적으로 관리함.
* **`kube-proxy`**
  * 노드로 들어오는 네트워크 트래픽의 라우팅을 담당함.
  * 주로 호스트의 `iptables` 또는 `IPVS`를 조작하여 가상 IP(Service) 체계를 구현함. 백엔드 개발자가 체감하는 내부 로드 밸런싱의 실체임.

# 2장: 로컬 쿠버네티스 환경 구축 (Local Kubernetes Setup)

## 1. `kubectl` 및 Kubeconfig 구조 분석
* **`kubectl`의 본질**
  * 단순 CLI 도구가 아니라 `kube-apiserver`로 HTTP REST 요청을 날리는 **REST 클라이언트**임.
  * 예: `kubectl get pods` 실행 시 내부적으로 `GET /api/v1/pods` API 호출로 변환됨.
* **Kubeconfig (`~/.kube/config`) 구조**
  * 여러 클러스터(Dev, Staging, Prod)를 안전하게 전환하며 제어하기 위한 설정 명세서임.
  * **Clusters:** 대상 API 서버 주소(URL) 및 CA 인증서 정보.
  * **Users:** 인증을 위한 토큰 또는 클라이언트 인증서 키 세트.
  * **Contexts:** 특정 User와 특정 Cluster를 매핑한 가상 환경 단위임. 배포 환경 오접속(운영 환경에 테스트 배포 등) 대형 사고를 막기 위해 컨텍스트 확인 습관이 필수적임.

## 2. 로컬 클러스터 도구 비교 (개발 프로세스 최적화)
* **Docker Desktop / Rancher Desktop 내장 K8s**
  * 별도 VM 설정 없이 토글 하나로 켤 수 있어 가장 직관적이고 가벼움.
  * 로컬 Docker 엔진과 네트워크를 공유하므로, 로컬 빌드 이미지를 Registry 푸시 없이 바로 Pod에 띄워 테스트하기 좋음.
* **Minikube**
  * 전통적인 단일 노드 클러스터 도구로, 다양한 드라이버와 인그레스(Ingress) 등의 내장 플러그인 활성화가 간편함.
* **Kind (Kubernetes in Docker)**
  * Docker 컨테이너 자체를 쿠버네티스 노드로 활용하는 '컨테이너 기반 클러스터'임.
  * 로컬에서 멀티 노드(Multi-node) 클러스터 시뮬레이션이 가능함.
  * 가볍고 부팅이 빨라 로컬 디버깅 외에도 **CI/CD 파이프라인(GitHub Actions 등)에 녹여 통합 테스트 자동화**할 때 가장 강력함.

## 3. API 그룹 및 버전 관리 구조
* 쿠버네티스 Manifest YAML 작성 시 상단에 정의하는 명세 규칙임.
* 기능 성격(Core, Apps, Networking 등)에 따라 API 단위를 모듈화하고, 성숙도(`alpha` -> `beta` -> `v1`)에 따라 버전을 관리함.
* 백엔드 설계 시 API 버전(`v1`, `v2`)을 나누어 하위 호환성을 유지하는 아키텍처 모델과 동일한 철학임.
