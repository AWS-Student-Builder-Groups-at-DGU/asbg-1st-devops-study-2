## 하이퍼바이저: 1형 vs 2형

하이퍼바이저는 하나의 물리 서버 위에서 여러 가상 머신을 생성하고 관리하는 소프트웨어다. 설치되는 위치에 따라 두 유형으로 나뉜다.

<img width="1158" height="870" alt="image" src="https://github.com/user-attachments/assets/2d31a227-3fc2-4815-92db-1551f7f2c90a" />


- **1형 (베어메탈)**: 하드웨어에 직접 설치된다. 중간 계층이 없어 오버헤드가 적고 성능이 우수하며, 주로 서버·데이터센터에서 쓰인다. VMware ESXi, Microsoft Hyper-V, KVM, Xen.
- **2형 (호스트형)**: 호스트 OS 위에 응용 프로그램처럼 설치된다. 설치·사용이 간편하지만 계층이 하나 더 껴서 오버헤드가 크다. VirtualBox, VMware Workstation, Parallels.

정리하면 1형은 성능 중심의 서버용, 2형은 편의성 중심의 개발용이다.

## 하이퍼바이저 vs 리눅스 컨테이너

가상 머신은 게스트 OS 전체를 가상화하고, 컨테이너는 호스트 커널을 공유한다.

| 구분 | 가상 머신 (하이퍼바이저) | 리눅스 컨테이너 (도커) |
| --- | --- | --- |
| 격리 방식 | 게스트 OS 전체를 가상화 | 호스트 커널 공유 + namespace·cgroups 격리 |
| 커널 | 가상 머신마다 독립 커널 | 호스트 커널을 공유 |
| 부팅 | OS 부팅이 필요해 느림 | 프로세스 실행 방식이라 빠름 |
| 크기 | GB 단위, 무겁다 | MB 단위, 가볍다 |
| 격리 강도 | 강함 | 상대적으로 약함 (커널 공유) |
| 오버헤드 | 큼 | 작음 |

컨테이너는 OS를 새로 기동할 필요가 없어 빠르고 가볍다. 대신 커널을 공유하는 만큼 격리 강도는 VM보다 다소 약하다.

## 도커 컨테이너란: namespace와 cgroups

도커 컨테이너는 도커 이미지를 기반으로 실행되는 프로세스다. 어디서 실행해도 같은 결과를 내며, 이를 Build Once, Run Anywhere라고 한다.

커널을 공유하는데도 컨테이너끼리 격리되는 이유는 리눅스의 두 기능 덕분이다.

<img width="756" height="915" alt="image" src="https://github.com/user-attachments/assets/8d9ec378-a20c-43bb-affc-18bc1a5a8b81" />


- **namespace**: 프로세스가 *무엇을 볼 수 있는가* 를 분리한다. PID, 네트워크, 파일시스템 등을 컨테이너마다 독립된 뷰로 제공해 다른 컨테이너의 자원이 보이지 않게 한다.
- **cgroups**: 프로세스가 *얼마나 쓸 수 있는가* 를 제한한다. CPU, 메모리 등 자원의 상한을 설정해 사용량을 제어한다.

namespace는 격리, cgroups는 자원 제한을 담당한다. 이 둘 덕분에 커널을 공유하면서도 독립된 OS처럼 동작할 수 있다.

## 컨테이너 설계 4원칙

이미지를 만들 때 지켜야 할 네 가지 기준이다.

| 원칙 | 내용 |
| --- | --- |
| 1 컨테이너 1 프로세스 | 한 이미지에 여러 프로세스를 올리지 않는다. 도커 생태계 도구들이 이 사상을 전제로 동작한다. |
| 변경 불가능한 인프라 (Immutable) | 실행에 필요한 리소스는 이미지에 포함한다. 기동 후 외부에서 설치·다운로드하는 방식을 피한다. |
| 경량 이미지 | 캐시 삭제, 경량 배포판(Alpine·Distroless), 멀티 스테이지 빌드로 크기를 최소화한다. |
| 비root 실행 계정 | root로 프로세스를 띄우면 보안 사고 위험이 크다. 전용 계정으로 권한을 최소화한다. |

한마디로 *가볍고, 변경되지 않으며, 안전하게* 설계한다.

## 기반 이미지와 크기

노드에 이미지가 없으면 외부에서 풀(pull)해야 하므로 이미지는 작을수록 유리하다. 같은 앱을 담아도 기반 이미지에 따라 40배 이상 크기가 차이 난다.

| 이미지 | 대략 크기 | 특징 |
| --- | --- | --- |
| scratch | ~0 | 아무것도 없는 정의용. 셸이 없어 디버그가 어렵다. |
| alpine | ~5.6MB | musl libc + busybox 기반 경량 리눅스. glibc와 동작 차이에 주의. |
| distroless | ~17MB | 특정 런타임(Python·Node.js 등)만 포함한 경량 대안. |
| ubuntu | ~73MB | 범용 배포판. 크지만 익숙하고 도구가 풍부하다. |
| centos | ~209MB | 범용 배포판. |
| UBI (Red Hat) | ~205MB | 기술 지원이 필요한 환경에 추천. |

## 도커 파일 명령어와 ENTRYPOINT vs CMD

이미지를 정의하는 주요 명령어다.

| 명령 | 요약 |
| --- | --- |
| `FROM` | 기반 이미지 지정 |
| `MAINTAINER` | 관리자 정보 (비추천, `LABEL` 사용) |
| `LABEL` | 메타데이터를 키:값으로 지정 |
| `USER` | 명령어 실행 계정 지정 |
| `WORKDIR` | 작업 디렉터리 지정 (없으면 생성) |
| `EXPOSE` | Listen 포트 지정 |
| `COPY` | 로컬 파일을 컨테이너로 복사 |
| `ADD` | tar.gz 압축을 풀어 복사 |
| `RUN` | 컨테이너 안에서 명령어 실행 |
| `ENTRYPOINT` | 기동 시 실행할 명령어 |
| `CMD` | 기동 시 실행할 명령어 인수 |

`ENTRYPOINT`와 `CMD`는 결합되어 하나의 실행 명령을 구성한다.

| ENTRYPOINT | CMD | 실행 결과 |
| --- | --- | --- |
| `"/bin/ls"` | `"-a"` | `/bin/ls -a` |
| `"/bin/ls"` | 지정 안 함 | `/bin/ls` |
| 지정 안 함 | `/bin/ls` | `/bin/ls` |
| `"/bin/sh","-c"` | `"ls -a"` | `/bin/sh -c "ls -a"` |

**실무 패턴**: `ENTRYPOINT`에 바뀌지 않는 기본 명령을, `CMD`에 기본 인수를 둔다. 실행 시 옵션을 주면 `CMD`를 덮어쓸 수 있다. 예를 들어 `ENTRYPOINT=/bin/sleep`, `CMD=3600`이면 기본은 `sleep 3600`이고 실행할 때 시간만 바꿔 쓸 수 있다.

## 도커 파일 예시

HTTP 요청에 `Hello, Kubernetes`로 응답하는 Go 앱 이미지다.

```dockerfile
FROM golang:1.14.1-alpine3.11      # 기반 이미지 지정
COPY ./main.go ./                  # 소스를 컨테이너로 복사
RUN go build -o ./go-app ./main.go # 빌드 시점에 컨테이너 안에서 실행
USER nobody                        # 실행 계정을 비root로 지정
ENTRYPOINT ["./go-app"]            # 기동 시 실행할 명령
```

```bash
docker image build -t sample-image:0.1 .
```

각 명령은 Step 단위로 실행되며 중간 컨테이너가 생성·삭제되면서 이미지가 만들어진다.

## 멀티 스테이지 빌드

`golang` 이미지에는 컴파일 도구가 300MB 이상 포함되어 최종 이미지가 377MB까지 커진다. 컴파일 도구는 빌드 시점에만 필요하고 실행에는 쓰이지 않는다. 멀티 스테이지 빌드는 이 문제를 해결한다.

<img width="1013" height="815" alt="image" src="https://github.com/user-attachments/assets/8d87815b-e943-4557-aa8c-6eb879ec0eac" />


```dockerfile
# Stage 1  컴파일
FROM golang:1.14.1-alpine3.11 as builder
COPY ./main.go ./
RUN go build -o /go-app ./main.go

# Stage 2  실행 이미지
FROM alpine:3.11
COPY --from=builder /go-app .
ENTRYPOINT ["./go-app"]
```

1단계에서 무거운 golang 이미지로 컴파일만 하고, 2단계에서 경량 alpine 이미지에 결과물만 복사한다. 컴파일 도구가 최종 이미지에서 빠지므로 **377MB → 13MB**로 줄고, scratch를 쓰면 약 7.4MB까지 축소된다. 크기가 줄 뿐 아니라 불필요한 도구가 없어 보안에도 유리하다.

## 레이어 통합과 축소화

이미 만든 이미지를 더 가볍게 만드는 방법이다.

<img width="1069" height="815" alt="image" src="https://github.com/user-attachments/assets/54cde57d-3a48-4418-9400-d4093d02ce9e" />


- **Dive**: OCI 표준 이미지를 레이어별로 조사하는 도구. 어느 파일 변경이 어느 정도 용량을 쓰는지 CUI에서 확인해 낭비 지점을 찾는다.
- **squash (`--squash`)**: 같은 파일이 여러 레이어에서 반복 변경될 때, 최종 상태만 가진 하나의 레이어로 통합해 이미지를 축소한다. (도커 실험적 기능)

## 도커 레지스트리

빌드한 이미지를 보관·배포하는 저장소 서버다. 대표적으로 Docker Hub, GCR(Google Container Registry), Amazon ECR이 있다.

<img width="943" height="518" alt="image" src="https://github.com/user-attachments/assets/6a98c7c7-2b9e-44c0-850d-814f14df05b3" />


이미지 이름은 다음 형식을 따른다. Docker Hub를 쓰면 호스트명을 생략하고 네임스페이스에 사용자 이름을 넣는다.

```
호스트명 / 네임스페이스 / 저장소 : 태그
```

```bash
docker login
docker image tag sample-image:0.1 USER/sample-image:0.1
docker image push USER/sample-image:0.1
docker logout
```

공식 이미지라도 취약점이 있을 수 있으므로 Trivy나 Clair 같은 도구로 스캔하는 것이 좋다.

## 전체 흐름

도커 파일 작성부터 컨테이너 기동까지의 전체 파이프라인이다.

<img width="951" height="535" alt="image" src="https://github.com/user-attachments/assets/e6a9e61e-d0f6-4966-bd2a-0d25a23a3f9a" />


```bash
# 컨테이너 기동 (localhost:12345 → 컨테이너 8080/TCP)
docker container run -d -p 12345:8080 sample-image:0.1
curl http://localhost:12345
# Hello, Kubernetes
```

`Dockerfile 작성 → 이미지 빌드 → 태그 → 레지스트리 push → 노드에서 pull → 컨테이너 기동`

쿠버네티스 환경에서는 마지막 기동을 쿠버네티스가 담당한다. 따라서 개발자가 맡는 부분은 가볍고 변경되지 않으며 안전한 이미지를 만들어 레지스트리에 올려두는 단계까지이며, 여기가 쿠버네티스 학습의 출발점이다.
