# 1장 도커 복습과 Hello, Kubernetes (Docker 복습)

## 1.1 도커 복습

쿠버네티스를 사용할 때 알아 두면 좋은 도커 지식은 크게 네 가지다.

- 도커 컨테이너 설계
- 도커 파일(Dockerfile) 작성법
- 도커 이미지 빌드
- 도커 레지스트리로 이미지 푸시(업로드)

### 도커 컨테이너란?

- 도커 컨테이너 = **도커 이미지를 기반으로 실행되는 프로세스**.
- 이미지만 있으면 환경에 상관없이 어디서든 동일하게 실행되어 **이식성이 높다**. 자바의 "Write Once, Run Anywhere"에 빗대어 컨테이너는 **"Build Once, Run Anywhere"**로 볼 수 있다.</br>
 <img width="337" height="187" alt="image" src="https://github.com/user-attachments/assets/18f4fef1-5c86-47e2-b878-de6b832dc0a0" />

- **가상 머신(VM) 대비 장점**: 경량이며 시작·중지가 빠르다.
  - VM: 하이퍼바이저 위에서 **게스트 OS**를 통째로 구동 → 부팅 대기 필요.
  - 컨테이너: **호스트 머신의 커널을 공유**하면서 **네임스페이스(격리)** 와 **cgroups(자원 제어)** 로 독립된 OS 같은 환경을 만든다 → 게스트 OS 부팅을 기다릴 필요가 없어 빠르다.

### 도커 컨테이너 설계 — 4가지 원칙

1. **1 컨테이너당 1 프로세스**
   - 하나의 이미지에 여러 프로세스를 올리지 않는다. 쿠버네티스 및 주변 에코시스템이 이 원칙을 전제로 만들어졌기 때문에, 어기는 경우 관리가 어려워질 수 있다.
2. **변경 불가능한 인프라(Immutable Infrastructure)로 만든다**
   - "환경을 바꿀 땐 기존 것을 없애고 새로 만든다", "한 번 만든 환경은 절대 수정하지 않는다"는 개념.
   - 컨테이너 기동 후 외부에서 바이너리/패키지를 가져오면 실행 결과가 달라질 수 있으므로, 컨테이너 기동전 **실행 바이너리·리소스를 이미지 안에 최대한 포함**시켜 불변 상태로 만든다.
   <img width="449" height="215" alt="image" src="https://github.com/user-attachments/assets/d28028da-eac1-4125-be11-9b8856de68c4" />
3. **경량 이미지로 만든다**
   - 노드에 이미지가 없으면 외부에서 풀(pull)해야 하므로 가벼울수록 좋다.
   - 방법: 패키지 설치 후 캐시 삭제, **멀티 스테이지 빌드**, 경량 배포판(**Alpine Linux**, **Distroless**), 레이어 축소/squash 등.
4. **실행 계정 권한 최소화**
   - 특히 **root 사용자 사용은 보안 사고로 이어질 수 있으므로** 되도록 피한다.

### 도커 파일(Dockerfile) 작성법

도커 이미지는 **Dockerfile**을 기반으로 빌드된다. Dockerfile은 "이미지 작성 매뉴얼"에 해당한다.

**예시 시나리오**: HTTP 요청이 오면 `Hello, Kubernetes`로 응답하는 Go 애플리케이션 이미지.

```dockerfile
# 기반 이미지 지정
FROM golang:1.14.1-alpine3.11
# 로컬 파일을 컨테이너로 복사
COPY ./main.go ./
# 빌드 시 컨테이너 내부에서 명령 실행
RUN go build -o ./go-app ./main.go
# 실행 계정 지정
USER nobody
# 컨테이너 기동 시 실행할 명령
ENTRYPOINT ["./go-app"]
```

**기반 이미지(Base Image) 비교**

| 이미지 | 특징 / 크기 |
|---|---|
| `scratch` | 최소(아무것도 없음). 셸도 없어 컨테이너 단독 생성 불가, 도커가 제공하는 사전 정의 이미지 |
| `alpine` | 작음. musl libc + busybox 기반 경량 리눅스. musl은 glibc와 동작 차이가 있어 주의 |
| `distroless` | 작음. Python/Node.js 등 특정 런타임만 포함 |
| `ubuntu` / `centos` | 큼 |
| `Universal Base Image(UBI)` | 큼. Red Hat 지원이 필요할 때 추천 |

**주요 명령 요약 (표 1-3)**

| 명령 | 요약 |
|---|---|
| `FROM` | 기반 이미지 지정 |
| `MAINTAINER` | 관리자 정보(현재 비추천 → `LABEL` 사용) |
| `LABEL` | 메타데이터를 key:value로 지정 |
| `USER` | 명령어 실행 계정 지정 |
| `WORKDIR` | 작업 디렉터리 지정(없으면 생성) |
| `EXPOSE` | 컨테이너가 Listen할 포트 지정 |
| `COPY` | 로컬 파일을 컨테이너로 복사 |
| `ADD` | tar.gz 등을 풀어서 복사 |
| `RUN` | 빌드 중 컨테이너 안에서 명령 실행 |
| `ENTRYPOINT` | 컨테이너 기동 시 실행할 명령 |
| `CMD` | 기동 시 실행할 명령의 인수(기본값) |

**ENTRYPOINT vs CMD**
- 실제 실행되는 것은 대략 `$ENTRYPOINT $CMD`.
- 관례: `ENTRYPOINT`에 바뀌지 않을 부분을, `CMD`에 기본 인수를 정의. 기동 시 인수만 덮어써서 실행할 수 있다.
  - 예: `ENTRYPOINT ["/bin/sleep"]`, `CMD`로 대기 시간(초)만 바꿔가며 사용.

### 도커 이미지 빌드

```bash
# -t 로 이름:태그 지정
$ docker image build -t sample-image:0.1 .
# 확인
$ docker image ls
```

**멀티 스테이지 빌드 (Multi-stage Build)**
- 문제: `golang` 이미지는 Go 컴파일 도구(약 364MB)를 포함 → 결과 이미지가 매우 커짐(예: 377MB).
- 해결: **여러 단계를 거쳐 처리하고, 최종 결과물만 실행용 이미지에 복사**.

```dockerfile
# Stage 1: 컴파일용
FROM golang:1.14.1-alpine3.11 as builder
COPY ./main.go ./
RUN go build -o /go-app ./main.go

# Stage 2: 실행용(결과 바이너리만 복사)
FROM alpine:3.11
COPY --from=builder /go-app .
ENTRYPOINT ["./go-app"]
```
<img width="402" height="237" alt="image" src="https://github.com/user-attachments/assets/820799b8-cfa2-418b-93b7-d1338cfc97d0" />

- 효과: 이미지 크기 대폭 감소(예: 377MB → **약 13MB**, `scratch` 기반이면 **약 7MB**).
- 장점: 컴파일 도구가 실행용 컨테이너에 남지 않아 **보안에 유리**하고, **BuildKit** 등으로 빌드 단계를 병렬 처리해 빌드 속도도 높일 수 있다.
- 3단계 이상 구성도 가능하며, `-f` 옵션으로 기본값인 Dockerfile이라는 이름의 파일이 아닌 사용할 파일의 이름을 지정할 수 있다.
```bash
$ docker image build -t sample-image:0.2 -f Dockerfile-multistage
```

### 도커 레지스트리로 이미지 업로드

- **도커 레지스트리** = 도커 이미지를 보관하는 저장소 서버.
  - 대표: **Docker Hub**, **Google Container Registry(GCR)**, **Amazon ECR** 등.
- Docker Hub에는 공개 이미지가 많음 → **공식 이미지(Docker Official Images)** 사용을 권장하고, 신뢰할 수 있는 이미지인지 확인.
- 공식 이미지라도 보안 패치가 안 된 취약점이 있을 수 있으므로, **Trivy / Clair** 같은 컨테이너 보안 스캔 도구 사용 검토.

**이미지 이름 형식**
```
[레지스트리 호스트명]/[네임스페이스]/[저장소]:[태그]
```
- Docker Hub 사용 시 호스트명은 생략 가능하며, 네임스페이스는 Docker Hub 사용자명을 쓴다.

**푸시 흐름**
```bash
$ docker login                                   # 로그인
$ docker image tag sample-image:0.1 USER/sample-image:0.1   # 태그 변경
$ docker image push USER/sample-image:0.1        # 업로드
$ docker logout                                  # 로그아웃
```

### 컨테이너 기동

```bash
# sample-image:0.1 이미지를 이용해 컨테이너 기동
$ docker container run -d -p 12345:8080 sample-image:0.1
# Hello, Kubernetes 출력 (컨테이너 실행 확인)
$ curl http://localhost:12345

```
- 단, **쿠버네티스를 사용할 때는 쿠버네티스가 컨테이너를 대신 기동**하므로 이 명령을 직접 쓰지는 않는다.

## 핵심 정리

- 컨테이너는 **호스트 커널 공유 + 네임스페이스/cgroups**로 격리된 프로세스 → VM보다 가볍고 빠르다.
- 컨테이너 설계 4원칙: **① 1컨테이너 1프로세스 ② 불변 인프라 ③ 경량 이미지 ④ 비-root 실행**.
- 이미지는 **Dockerfile**로 빌드하고, **멀티 스테이지 빌드**로 크기·보안을 개선한다.
- 이미지는 **레지스트리**에 push/pull할 수 있으며 공개 이미지 사용시 보안을 확인해야 한다.
