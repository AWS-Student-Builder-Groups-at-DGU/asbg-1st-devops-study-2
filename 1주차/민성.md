# 1장. 도커 복습과 Hello, Kubernetes

> 『쿠버네티스 완벽 가이드』 1주차 정리 — 민성

쿠버네티스는 "컨테이너를 대규모로 운영하기 위한 도구"다. 그래서 쿠버네티스를 배우기 전에 그 대상인 **컨테이너(도커)** 를 먼저 복습하는 것이 1장의 목적이다. 흐름은 `도커 개념 → 이미지 빌드 → 레지스트리 업로드 → 컨테이너 기동 → 왜 쿠버네티스가 필요한가`로 이어진다.

---

## 1.1 도커 복습

### 컨테이너란?
- 호스트 OS의 커널을 공유하면서, 애플리케이션과 그 실행 환경(라이브러리, 설정 등)을 **하나로 묶어 격리된 프로세스**로 실행하는 기술.
- 가상 머신(VM)이 게스트 OS 전체를 통째로 올리는 것과 달리, 컨테이너는 커널을 공유하므로 **가볍고(용량↓) 빠르게(기동↑)** 뜬다.

| 구분 | 가상 머신(VM) | 컨테이너 |
| --- | --- | --- |
| 격리 단위 | 게스트 OS 전체 | 프로세스 |
| 커널 | 게스트마다 별도 | 호스트 커널 공유 |
| 기동 속도 | 느림(수십 초~분) | 빠름(초 단위) |
| 용량 | 큼(GB) | 작음(MB) |
| 이식성 | 낮음 | 높음("어디서나 동일하게 실행") |

### 1.1.1 도커 컨테이너란?
- 리눅스 커널의 **namespace(자원 격리)** 와 **cgroups(자원 제한)** 기능을 활용해 프로세스를 격리한다.
  - namespace: PID, 네트워크, 마운트, 사용자 등을 컨테이너별로 분리
  - cgroups: CPU·메모리 등 자원 사용량을 제한
- **이미지(Image)**: 컨테이너를 실행하기 위한 "읽기 전용 템플릿". 여러 개의 **레이어(layer)** 가 쌓여 구성되며, 레이어는 캐시되어 재사용된다.
- **컨테이너(Container)**: 이미지를 실행한 "실행 중인 인스턴스". 이미지 위에 쓰기 가능한 레이어가 얹힌다.

> 핵심: **이미지는 설계도, 컨테이너는 그 설계도로 만든 실체.** 하나의 이미지로 여러 컨테이너를 띄울 수 있다.

## 1.2 도커 이미지 만들기 (Dockerfile & build)

### 도커 파일(Dockerfile) 작성법
컨테이너 이미지를 코드로 정의하는 파일. 대표 명령어:

| 명령어 | 역할 |
| --- | --- |
| `FROM` | 베이스 이미지 지정 (예: `FROM ubuntu:20.04`) |
| `RUN` | 빌드 시점에 실행할 명령 (패키지 설치 등) |
| `COPY` / `ADD` | 파일을 이미지 안으로 복사 |
| `WORKDIR` | 작업 디렉터리 지정 |
| `ENV` | 환경 변수 설정 |
| `EXPOSE` | 컨테이너가 사용할 포트 명시(문서 목적) |
| `CMD` | 컨테이너 기동 시 기본 실행 명령 (덮어쓰기 가능) |
| `ENTRYPOINT` | 컨테이너 기동 시 고정 실행 명령 |

예시 Dockerfile:
```dockerfile
FROM nginx:1.19
COPY index.html /usr/share/nginx/html/
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

### 베이스 이미지
용도에 따라 골라 쓴다. 가벼울수록 보안·용량에 유리.
- `scratch` : 아무것도 없는 빈 이미지(최소)
- `alpine` : 매우 가벼운 리눅스(수 MB)
- `distroless` : 셸·패키지 매니저 없이 앱 실행에 필요한 것만
- `busybox`, `ubuntu`, `centos`, Universal Base Image(UBI) 등

### 이미지 빌드
```bash
# -t 로 "이름:태그" 지정, 마지막 . 은 빌드 컨텍스트(현재 폴더)
docker build -t myapp:1.0 .

docker images        # 만들어진 이미지 확인
```
- `CMD` vs `ENTRYPOINT`: ENTRYPOINT는 항상 실행되는 고정 명령, CMD는 기본 인자로 실행 시 덮어쓸 수 있다. 둘을 조합해서도 쓴다.

## 1.3 레지스트리로 이미지 업로드

- **레지스트리(Registry)**: 이미지를 저장·공유하는 저장소. Docker Hub(공개), GCR/ECR/ACR(클라우드), 사내 프라이빗 레지스트리 등.
- 이미지 이름 규칙: `레지스트리/사용자(리포지터리)/이미지명:태그`
```bash
docker login
docker tag myapp:1.0  myname/myapp:1.0     # 레지스트리용 이름 붙이기
docker push myname/myapp:1.0               # 업로드
docker pull myname/myapp:1.0               # 내려받기
```
> 쿠버네티스가 컨테이너를 띄우려면 이미지를 레지스트리에서 pull 하므로, "이미지를 레지스트리에 올려두는 것"이 실무의 기본 전제다.

## 1.4 컨테이너 기동

```bash
docker run -d -p 8080:80 --name web nginx:1.19
```
- `-d` : 백그라운드(detached) 실행
- `-p 8080:80` : 호스트 8080 → 컨테이너 80 포트 매핑
- `--name` : 컨테이너 이름 지정
- `-e KEY=VALUE` : 환경 변수 주입
- `-v host:container` : 볼륨(데이터 영속화) 마운트

자주 쓰는 조작:
```bash
docker ps              # 실행 중 컨테이너 목록 (-a: 정지 포함)
docker logs web        # 로그 확인
docker exec -it web sh # 컨테이너 안으로 진입
docker stop web        # 정지
docker rm web          # 삭제
```

## 1.5 쿠버네티스의 세계로

도커만으로 컨테이너 하나하나는 잘 띄운다. 하지만 **여러 서버에 걸쳐 수십·수백 개 컨테이너를 운영**하려 하면 문제가 생긴다.

도커 단독으로는 어려운 것들:
- 컨테이너가 죽으면 자동으로 다시 살리기 (자동 복구)
- 여러 노드(서버)에 컨테이너를 알아서 배치하기 (스케줄링)
- 부하에 따라 개수 늘리고 줄이기 (오토 스케일링)
- 트래픽을 여러 컨테이너에 분산하기 (로드 밸런싱 / 서비스 디스커버리)
- "원하는 상태"를 코드로 선언하고 유지하기 (선언적 관리)
